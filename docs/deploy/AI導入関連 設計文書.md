## 1. プロジェクトの背景と目的

### 1.1 概要

社内AI利用プロジェクトの一環として、非IT職約20名がVSCode + Claude Codeを使ったバイブコーディングで業務アプリ（PoC）を作成する。作成したアプリを簡単にデプロイして動作確認できる環境を、社内ポリシーに準拠した形で用意する。

### 1.2 最上位の要求仕様

**「git pushでデプロイされること」** — 可能な限りVercel/Cloudflareの無料プランのデプロイ体験をAzure上で再現する。管理者側の運用負荷も同様に最小化する。

### 1.3 前提条件

| 項目 | 内容 |
|---|---|
| 開発者 | 非IT職 約20名 |
| アプリ数 | 20〜30程度（随時増減） |
| 利用者規模 | 各アプリ 数名〜数十名 |
| 用途 | PoC（有用と判断されたものは別途正式環境を検討） |
| 社内ポリシー | データ海外保持禁止、学習等へのデータ利用禁止 |
| クラウド標準 | Azure（既存利用実績あり） |
| 技術スタック | Node.js / React に基本統一 |

### 1.4 本基盤のスコープ

本基盤は、汎用的に使えるVSCode + Claude Codeという環境をWebアプリ開発に使う場合の、セキュアで企業ポリシーに合致したPoC環境を提供するものである。Claude Code自体のライセンスコストや利用管理は本基盤のスコープ外とする（Claude Codeは開発専用ではなく汎用ツールであるため、別枠で管理する）。

## 2. アーキテクチャ

### 2.1 構成図

```
Subscription
 └ RG: poc-platform (Japan East)
     ├ VNet (poc-vnet)
     │   ├ Subnet: app-integration   ← App Service VNet統合用
     │   └ Subnet: private-endpoints ← Private Endpoint用
     ├ App Service Plan (P0v3 / Linux) ── VNet統合 → app-integration
     │   ├ Web App (12345-attendance-checker)
     │   ├ Web App (23456-sales-dashboard)
     │   └ Web App (xxxxx-app-name) ... 最大20〜30アプリ
     ├ Cosmos DB Account (Serverless)  ← Private Endpoint経由のみ（パブリックアクセス無効）
     ├ Storage Account (Blob)          ← Private Endpoint経由のみ（パブリックアクセス無効）
     ├ Private DNS Zone (privatelink.documents.azure.com)
     ├ Private DNS Zone (privatelink.blob.core.windows.net)
     └ Entra ID アプリ登録（共有・1つ） ← Easy Auth用
```

### 2.2 認証方式

Easy Auth（App Service組み込み認証）を使用し、Entra IDによる社内認証を全アプリに適用する。

- 全アプリで1つのEntra IDアプリ登録を共有する
- プロビジョニング時に新しいWeb AppのリダイレクトURIを自動追加
- **社外ネットワーク（出先・スマートフォン等）からもWebアプリにアクセス可能。ただしEntra IDでの認証が必須であり、社員アカウントでMicrosoftログインしなければアクセスできない。社員以外はアクセス不可**
- アプリ単位のアクセス制御はPoC段階では不要と判断（社員であればアクセス可能）
- 特定アプリのみアクセス制限が必要な場合は個別対応

**制約事項:** Entra IDアプリ登録のリダイレクトURIは256個が上限。20〜30アプリでは問題ないが、大幅なスケール拡大時にはアプリ登録の分割が必要になる。

### 2.3 ネットワーク統制

機密性の高いデータ（顧客情報・個人情報等）を扱う可能性があるため、バックエンドリソースをネットワークレベルで閉域化する。

**バックエンド（Cosmos DB / Blob Storage）：**

- App Service PlanにリージョンVNet統合を設定し、バックエンドへの通信をVNet経由に限定
- Cosmos DB・Blob StorageそれぞれにPrivate Endpointを作成
- パブリックアクセスを無効化し、VNet内からのみ接続可能とする
- Private DNSゾーン（`privatelink.documents.azure.com`, `privatelink.blob.core.windows.net`）をVNetにリンク
- 環境変数（キー・接続文字列）がコードのミスで漏洩した場合でも、インターネット経由でのアクセスをネットワーク層で遮断

**初期構築時の例外:** 初期構築時はCosmos DB・Blob Storageに業務データが存在しないため、管理者IPアドレスからのパブリックアクセスを一時的に許可する。キー取得・初期設定完了後にパブリックアクセスを無効化する。手順の詳細は管理者ランブックを参照。

**フロントエンド（Web App）：**

- IPアドレスによるアクセス制限は設けない。営業スタッフの出先からのモバイルアクセス等を考慮し、ネットワーク制限は現実的でないと判断
- Easy Auth（Entra ID認証）によりアクセスを社内ユーザーに限定（2.2項で定義）

**追加コスト：** Private Endpoint 2つで約$15/月。VNet・サブネット・Private DNSゾーンは無料。

### 2.4 デプロイフロー概要

開発者の日常操作は「Claude Codeにプッシュを指示する」のみ。アプリの初期作成はGoogle Form申請からの自動プロビジョニングで実施する。

**申請〜プロビジョニング（自動）：**

1. ユーザーがGoogle Formでアプリ物理名・論理名を申請
2. GASがGitHub Actionsをトリガー
3. GitHub Actions上でリポジトリ作成・Web App作成・Deployment Center接続・Easy Auth設定を自動実行
4. GAS経由で申請者に完了メール（リポジトリURL・デプロイ先URL）

**開発者の日常操作：**

1. `git clone` でローカルに取得
2. Claude Codeでアプリを開発・修正
3. ローカルで動作確認（`npm run dev`）
4. Claude Codeに「プッシュして」と指示（git push）
5. GitHub Actionsが自動実行 → 数分後に本番URL反映

詳細はプロビジョニング仕様書およびユーザーガイドを参照。

## 3. 技術スタック

### 3.1 基本スタック

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | React (Vite) | Claude Codeとの相性が良く、情報が豊富。Viteでビルドが高速 |
| バックエンド | Node.js (Express) | フロントと同一言語で完結。非IT職の学習コストを最小化 |
| データベース | Cosmos DB (Serverless) | 従量課金でPoCに最適。使わなければほぼ無料 |
| ファイル保存 | Azure Blob Storage | 安価でシンプル |
| 認証 | Entra ID (Easy Auth) | App Service組み込み。コード不要で社内認証を実現 |
| CI/CD | GitHub Actions | Deployment Centerにより自動生成 |

### 3.2 スタック統一の理由

- Claude Codeへのプロンプトテンプレート（CLAUDE.md）を共通化できる
- App Serviceのランタイム設定が統一され管理が容易
- メンバー間でノウハウを共有しやすい
- トラブルシューティングも共通パターンで対応可能

### 3.3 例外

- **許容する例外：** Python（Flask / FastAPI）のみ。管理者の事前承認が必要
- **想定ケース：** データ分析・機械学習等、Node.jsでは実現困難な処理がある場合に限る
- **対応方法：** 例外案件は自動プロビジョニングの対象外とし、管理者が手動対応。デプロイフロー（git push → GitHub Actions → 本番反映）は標準と同じ
- **上記以外のスタック（Java, .NET, PHP等）：** 原則不可
- **自動化への組み込み：** 例外の実績が一定数溜まった段階で、プロビジョニングパイプラインへのスタック選択パラメータ追加を検討

## 4. テンプレートリポジトリ

プロビジョニング時にテンプレートリポジトリの内容が新規リポジトリの初期状態としてコピーされる。

```
poc-app-template/
├── CLAUDE.md           # Claude Code用の指示書（技術スタック制約等）
├── .gitignore
├── package.json
├── server.js           # Express最小構成
├── public/
│   └── index.html      # Reactエントリポイント placeholder
├── src/
│   ├── App.jsx         # Reactルートコンポーネント placeholder
│   └── main.jsx        # Viteエントリ
└── vite.config.js
```

GitHub Actionsのワークフローファイルはテンプレートに含めない。Deployment Center接続時にAzure側が自動生成する。テンプレートの具体的な内容（CLAUDE.md、package.jsonの詳細）は別途検討の上で差し替える。

**検証事項:** Deployment Center自動生成ワークフローとVite + Express構成の互換性を、テンプレート設計時に必ず検証する。特に`npm run build`の出力先とExpressの静的ファイル配信パスの整合性。

## 5. 命名規則

### 5.1 アプリ物理名

**フォーマット：** `{社員番号}-{アプリ名}`

| 項目 | ルール |
|---|---|
| 使用可能文字 | 半角英小文字、半角数字、ハイフン |
| 文字数制限 | 全体30文字以内 |
| 社員番号部分 | 4〜6桁の数字 |
| アプリ名部分 | 英小文字・数字・ハイフンで構成。20文字以内 |
| 先頭にハイフン不可 | Azure Web App制約 |
| アンダースコア不可 | Azure Web App名で使用不可 |
| バリデーション正規表現 | `^[0-9]{4,6}-[a-z0-9]+(-[a-z0-9]+)*$`（全体30文字以内） |

先頭に社員番号を置くのは、一覧表示時にユーザーごとにソートされ管理しやすいため。

### 5.2 命名が使われる場所

| 場所 | 値 |
|---|---|
| GitHubリポジトリ名 | `{物理名}` |
| Azure Web App名 | `{物理名}` |
| デプロイURL | `https://{物理名}.azurewebsites.net` |
| Cosmos DBコンテナプレフィックス | `{物理名}-` |
| Blob Storageコンテナ | `{物理名}` |

## 6. Cosmos DB / Blob Storage

### 6.1 共通環境変数

全アプリに対し、プロビジョニング時に以下の環境変数を共通で設定する。

| 環境変数 | 用途 |
|---|---|
| `COSMOS_ENDPOINT` | Cosmos DBエンドポイントURL |
| `COSMOS_KEY` | Cosmos DB共有キー |
| `COSMOS_DATABASE` | 共有データベース名（`poc-db`） |
| `STORAGE_CONNECTION_STRING` | Blob Storage接続文字列 |
| `STORAGE_CONTAINER_PREFIX` | アプリ物理名（コンテナ分離用） |

### 6.2 データ分離

アプリ間のデータ分離はCosmos DBコンテナの命名規則（`{物理名}-{用途}`）およびBlob Storageのプレフィックスによる規約ベースの分離とする。PoC用途であるため、技術的な強制分離は行わない。

### 6.3 ローカル開発とデータアクセスの方針

- **ローカル開発時はローカルファイル（JSON/SQLite等）にデータを保存し、デプロイ時はCosmos DB/Blobに接続する**
- テンプレートにローカル/本番の自動切り替え機構は組み込まない
- 接続先の切り替え実装は開発者がClaude Codeに指示して自身で管理する
- データは必要な形に成形した上でアプリにアップロードする運用とし、アプリ側での変換処理を最小化する

**この方針の意図：** ローカル環境であえてJSONやCSVなどテキストベースのファイルを使用させることで、データ管理の違い（ファイルとDB）や、自分が必要とするデータの形式を意識させる教育的狙いがある。管理者への切り替えミス対応の依存を排除する運用上の意図も兼ねる。

## 7. GitHub Organization

### 7.1 基本方針

- GitHub Free Organizationを使用
- 社用メールで個人アカウントを作成しOrgに参加
- リポジトリはすべてOrg配下にprivateで作成
- リポジトリはアプリ単位で分離（1リポジトリ = 1アプリ）。複数アプリの混在禁止

リポジトリをアプリ単位で分離する理由：AI駆動開発ではワークスペース内ファイルがコンテキストになるため、無関係なアプリの混在は誤変更リスクがある。

### 7.2 GitHub Actions分数管理

GitHub Free Organizationのプライベートリポジトリでは月間2,000分がOrg全体で共有される。

**消費見積もり：** デプロイ1回あたり2〜5分。本基盤ではローカル環境で`npm run dev`によるテストが可能であり、ローカル環境ではローカルファイルでデータの保持・入出力を行う方針のため、頻繁にpushする必要性は薄い。

**対応方針：** Free枠の2,000分で運用する。spending limitは設定しない。実消費量をモニタリングし、上限に近づいた場合に以下を検討する。

- 従量課金での追加購入（$0.008/分。月500分超過で$4）
- GitHub Teamプラン（$4/user/月。3,000分に増加）
- セルフホストランナー（Azure VM B1sクラス 月$7〜10）

### 7.3 PRプレビュー

Vercel的なPRプレビュー機能は実装しない。個人のPoC用途でありPRレビュー運用は過剰。本番直接反映→問題あれば戻す運用で十分。

## 8. デプロイ認証方式の設計

本基盤では意図的に2つの認証方式を使い分ける。

| 用途 | 認証方式 | 理由 |
|---|---|---|
| プロビジョニング（GitHub Actions → Azure CLI） | OIDC federation（Workload Identity） | シークレットレスでセキュア |
| 個別アプリのデプロイ（GitHub Actions → App Service） | Publish Profile | Deployment Center自動生成ワークフローの標準方式 |

OIDC federationのsubjectフィルタは `repo:{org}/poc-platform-admin:ref:refs/heads/main` を設定する。プロビジョニングは`repository_dispatch`イベントで起動するが、`repository_dispatch`はデフォルトブランチ（main）上で実行されるため、このフィルタで動作する。万が一動作しない場合は、フェデレーション資格情報のsubjectを `repo:{org}/poc-platform-admin:*` に変更する。**初回構築時に必ず実機検証し、動作しない場合は即座にワイルドカードに変更する。**

## 9. プラットフォーム選定とその根拠

### 9.1 選定結果

| 決定事項 | 内容 | 理由 |
|---|---|---|
| Azure App Serviceを採用 | Vercel/Cloudflareは不採用 | 社内ポリシー準拠、コスト、管理一元性（後述の詳細比較参照） |
| App Service Plan P0v3 | S1/S2ではなくPremium v3 | S1/S2はレガシープラン。新規構築ではP0v3が適切 |
| Static Web Apps不採用 | SWAは使わない | GitHubリポジトリ更新低調、複数機能がプレビュー止まり、Dedicatedプラン廃止（2025年10月）。将来性に懸念 |
| Container Apps不採用 | 今回の要件では見送り | 非IT職にとってデプロイが複雑。ゼロスケール可能だが今回はApp Serviceで十分 |

### 9.2 Vercel環境との対比（詳細）

#### Vercel単体ではPoCアプリが成立しない

PoCアプリの中心機能はデータのCRUD（作成・読取・更新・削除）と集計・可視化であり、Vercel単体にはDB/ストレージ機能がなく、データの永続化手段を持たない。さらに、VercelのHobbyプラン（無料）は非商用・個人利用のみに制限されており、社員が業務時間中にPoCアプリを開発する行為は商用利用の定義に抵触する可能性が高い（Vercel Fair Use Guidelines）。

#### 外部DBを組み合わせると管理が分散しコストが増加する

DB/ストレージを外部サービス（Supabase等）で補完する場合、非IT職20名が個別にDBサービスのアカウント作成・リージョン選択・接続設定を行う運用は非現実的である。コスト面でもVercel Pro + Supabase構成は月額約$615で、Azure構成（$140〜160）の約4倍となる。

#### データ所在とポリシー準拠

| サービス | 日本リージョン | 備考 |
|---|---|---|
| Neon（PostgreSQL） | なし | 日本国内にデータを保持する手段がない |
| Supabase | あり（東京） | 顧客データは指定リージョンにのみ保存・処理される |
| Vercel管理データ | 日本に限定されない | ソースコード、ログ等は米国含む法域で処理される可能性 |
| Azure（本構成） | Japan East | 全リソースが日本リージョンに統一 |

#### 研修環境と本番環境の一貫性

Azure基盤を研修段階から使うことで、Vercel→Azureの環境移行による断絶を排除する。非IT職の参加者はコードの内部を深く理解していないため、環境の違いによるエラーを自力で解決することが極めて難しい。「研修で動いたのに本番で動かない」という体験は挫折に直結する。

#### Vercelの構造的不適合

Vercelは「プロの開発者の生産性を最大化する」ことを設計思想としている。非IT職が実際に助けを必要とする領域（DB接続、環境変数管理、サービス間の設定整合）は、Vercelが意図的にスコープ外としている部分である。この構造的な不適合は、プランの選択やコストの調整では解消できない。

#### Azure基盤の認識すべきリスク

- **構築者への依存：** 基盤の構築・保守が特定担当者に依存する。IaCと運用ドキュメントで軽減
- **障害対応の内部化：** 障害の切り分け・復旧が自社責任
- **デプロイ体験の差：** Vercelほどの手軽さは提供できないが、PoCの文脈では影響は軽微

#### 比較一覧

| 観点 | Vercel Hobby | Vercel Pro + Supabase | Azure基盤（本設計） |
|---|---|---|---|
| デプロイ体験 | 優秀（数十秒） | 優秀（数十秒） | 良好（2〜5分） |
| データ永続化 | 不可 | 可能 | Cosmos DB / Blob（自動設定） |
| 規約上の利用可否 | 商用利用禁止 | 商用利用可 | 問題なし |
| データ所在 | 米国含む | グレーゾーン（Vercel側） | Japan East統一 |
| 月額コスト（20名規模） | $0（規約違反リスク） | 約$615 | 約$140〜160 |
| 社内ポリシー準拠 | 不可 | グレーゾーン | 準拠 |
| 研修→本番一貫性 | 低い | 低い | 高い |
| 設計思想の適合 | 不適合 | 不適合 | 適合 |

## 11. 運用方針

### 11.1 運用期間と位置づけ

本基盤は恒久利用を前提とせず、まず数か月間（目安：3か月）の運用を行い、基盤自体の有効性を検証する。技術設計は恒久運用にも耐える水準で設計されている。PoC期間中に検証すべきは技術構成ではなく、運用コスト（金銭・人的）が継続的に負担可能かどうかである。

### 11.2 管理者の対応範囲

管理者は週に数時間のリソースを確保する。管理者が対応するのは基盤起因の問題に限定する。

**管理者が対応する事項：** プロビジョニングの失敗・手動リカバリ、Easy Authのエラー、App Service Planの容量・パフォーマンス問題、GitHub Actionsの分数管理・障害、アプリの削除・棚卸し、基盤構成の変更・メンテナンス

**管理者が対応しない事項（ユーザー自力解決）：** アプリ固有のビルドエラー・実行時エラー、デプロイ後のアプリ動作不良、Claude Codeの使い方に関する質問、アプリの機能設計・実装方法

### 11.3 ユーザーの自力解決支援

管理者が管理するGitHubリポジトリ（poc-docs等）にトラブルシュートドキュメントを配置する。ユーザーはClaude Codeを通じてこのドキュメントを参照し、自力で問題を解決する。各アプリのCLAUDE.mdにはトラブルシュート情報を直接記載せず、トラブルシュートドキュメントのURL参照のみを記載する。

### 11.4 RBAC

| 役割 | 権限 | 対象者 |
|---|---|---|
| プラットフォーム管理者 | App Service Plan / Cosmos DB / Storageの管理 | IT担当 1〜2名 |
| 開発者 | 自分のWeb Appの閲覧・ログ確認（デプロイはGitHub経由） | 非IT職 約20名 |

### 11.5 ライフサイクル管理

- 全Web Appに作成日タグを付与し、定期的に棚卸しを実施
- 一定期間（例：90日）アクセスのないアプリは削除候補としてリストアップ
- 恒久利用が有用と判断されたアプリは正式環境への昇格プロセス（11.6項）に進む

### 11.6 正式環境への昇格プロセス

**判断主体：** アプリ作成者の上長・利用部門が「恒久利用したい」と判断し、IT部門に移行を依頼。ビジネス判断として部門側が起点。

**コードの扱い：** PoCコードをそのまま正式環境に移行する。動作実績をもって品質とみなす。

**移行先環境：** 案件の規模・要件に応じて個別に決定（専用App Service、Container Apps等）。

**PoC基盤側のクリーンアップ：** Web Appの停止・削除、GitHubリポジトリのアーカイブ、Cosmos DB / Blob Storageの該当データの移行または削除、Entra IDアプリ登録からリダイレクトURIを削除、管理スプレッドシートのステータスを「昇格済み」に更新。

### 11.7 P0v3の容量限界

実測してから段階的に対応する：ノイジーネイバー特定 → Plan分割 → スケールアップ。

## 12. 評価と継続判断

### 12.1 評価時期

PoC運用開始から3か月後を目処に評価を実施する。

### 12.2 評価観点

| 観点 | 評価方法 |
|---|---|
| アプリ作成数 | プロビジョニング実績の集計 |
| 業務利用実績 | 実際に業務で使われているアプリの数と内容 |
| 管理者の実負荷 | 週あたりの対応時間・対応内容の記録 |
| トラブル頻度と内容 | 基盤起因 vs アプリ起因の分類と件数 |
| ユーザーの自力解決率 | トラブルシュートドキュメントで解決できた割合 |
| コスト実績 | Azure・Actions の実消費額 |

### 12.3 判断の選択肢

| 判断 | 内容 |
|---|---|
| 正式基盤化 | PoC基盤の運用実績を要件定義のインプットとし、正式な社内基盤として再設計 |
| 改善して継続 | PoC期間を延長し改善を実施 |
| 終了 | 基盤を停止しリソースを削除 |

### 12.4 成功時の方針（重要）

基盤が有効と評価された場合でも、PoC基盤をそのまま延命して恒久運用に移行しない。理由：

- PoC期間中は管理者の週数時間のリソースで手動リカバリを含めて凌ぐ前提であり、恒久運用に耐える運用設計ではない
- 仮設として作った基盤を恒久化する判断は組織的に通りやすい一方、恒久運用に耐える設計への改修コストは初期構築より高くなる

正式基盤化の判断がなされた場合は、PoC基盤の運用実績を要件定義のインプットとし、正式な社内PaaS基盤として要件定義からやり直す。**この方針はPoC開始前に意思決定者の合意を得ておくこと。**

## 13. 意思決定者への事前説明事項

PoC開始の承認を得る際に、以下を明示的に説明し合意を得ること。

1. **PoC期間のコスト：** 3か月で約6〜7万円（Azure基盤直接費用のみ。Claude Codeは別枠）
2. **恒久化した場合の年間コスト：** 約25〜30万円（Azure基盤直接費用のみ。人件費・追加投資は別途）
3. **PoC基盤は仮設である：** 成功しても延命せず、正式基盤として再設計する方針であること
4. **PoC開始の承認は、成功時のコスト負担を含めた意思決定であること**

## 14. リスクと注意事項

| リスク | 影響 | 対策 |
|---|---|---|
| ノイジーネイバー | 1アプリの重い処理が他に影響 | App Insightsで監視、問題アプリを特定・制限 |
| Plan容量不足 | 30アプリ超で性能低下 | P1v3へのスケールアップまたはPlan分割 |
| GitHub Actions分数超過 | デプロイが実行されない | Free 2,000分で運用。近づいたら従量課金等を検討 |
| コスト超過 | Cosmos DBの予想外の利用増 | 請求アラート設定、月次レビュー |
| AI駆動開発による誤変更 | Claude Codeが意図しないファイルを変更 | リポジトリ・フォルダをアプリ単位で分離 |
| 構築者への依存 | 担当者離任で運用困難 | IaCと本ドキュメント群で軽減 |
