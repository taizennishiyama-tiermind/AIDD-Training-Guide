# Claude Code 実装・デプロイ標準マニュアル

## 1. このドキュメントの目的

本ドキュメントは、非 IT 職の利用者が **VSCode + Claude Code** を用いて業務アプリを作成し、社内ポリシーに準拠した Azure 環境へ簡易にデプロイできるようにするための標準手順書である。
対象となるアプリは主に PoC 用途であり、恒久利用が必要になった場合は別途正式環境への移行を検討する。 

このドキュメントの目的は次の 3 点である。

1. Claude Code が生成するアプリの技術構成を標準化する
2. Azure の共有デプロイ環境に適合する実装ルールを明確化する
3. 最終的に Azure App Service 上へ安定してデプロイできる状態を作る

---

## 2. 前提条件

本プラットフォームは、以下の条件を前提として設計する。 

| 項目       | 内容                     |
| -------- | ---------------------- |
| 開発者      | 非 IT 職 約 20 名          |
| アプリ数     | 20～30 程度               |
| 利用者規模    | 各アプリ 数名～数十名            |
| 用途       | PoC                    |
| クラウド標準   | Azure                  |
| 基本技術スタック | Node.js / React        |
| 社内ポリシー   | データ海外保持禁止、学習等へのデータ利用禁止 |

---

## 3. 本環境で守るべき原則

### 3.1 社内ポリシー準拠

本環境では、以下を必須要件とする。 

* データを国内保持できること
* 学習やサービス改善目的でデータ利用されないこと
* 社内標準クラウドである Azure を利用すること

### 3.2 Claude Code による実装標準化

Claude Code は、アプリ生成時に毎回自由な構成を採用するのではなく、**必ず本ドキュメントの標準構成に合わせてアプリケーションを作成する**。
これにより、管理者運用、トラブル対応、再利用、教育コストを最小化する。 

### 3.3 デプロイ体験の統一

利用者の日常操作は原則として **`git push` のみ** とする。
Azure 側へのデプロイは GitHub Actions により自動実行される。 

---

## 4. 推奨アーキテクチャ

### 4.1 全体構成

Azure App Service Plan を 1 台用意し、その上に複数の Web App を載せる共有ホスティング構成とする。
リージョンは **Japan East** を採用する。 

```text
App Service Plan (P0v3 / Linux / Japan East)
  ├ Web App (project-a)  ← https://project-a.azurewebsites.net
  ├ Web App (project-b)  ← https://project-b.azurewebsites.net
  ├ Web App (project-c)
  └ Web App (project-n)

  + Cosmos DB (Serverless)  ← DBが必要なアプリ用
  + Blob Storage            ← ファイル保存が必要なアプリ用
```

### 4.2 採用理由

この構成を採用する理由は次の通りである。 

* データ国内保持要件に対応できる
* PoC 用途に対してコスト効率が高い
* `git push` ベースの簡易なデプロイ体験を実現できる
* Node.js / React ベースのアプリを複数運用しやすい
* 20～30 アプリ規模の PoC を共有基盤で扱える

## 5. Claude Code が必ず従う実装方針

この章は、Claude Code に対する実装ルールとして扱う。

## 5.1 必須技術スタック

Claude Code は、特別な例外が認められない限り、以下のスタックでアプリを生成する。 

| レイヤー    | 標準技術                   |
| ------- | ---------------------- |
| フロントエンド | React (Vite)           |
| バックエンド  | Node.js (Express)      |
| データベース  | Cosmos DB (Serverless) |
| ファイル保存  | Azure Blob Storage     |
| 認証      | Entra ID (Easy Auth)   |
| CI/CD   | GitHub Actions         |

### 5.2 生成するアプリの基本原則

Claude Code は次の方針で実装する。 

* フロントエンドは React + Vite を前提とする
* バックエンドは Node.js + Express を前提とする
* Node.js と React を中心に、利用者が理解しやすい構成にする
* PoC に不要な複雑構成は避ける
* DB が不要な場合は Cosmos DB を無理に使わない
* ファイル保存が不要な場合は Blob Storage を無理に使わない
* 認証が必要な場合は Entra ID の Easy Auth を前提とする
* Azure App Service で動作する前提で作る

### 5.3 実装時の最適化方針

Claude Code は、Azure App Service 上での運用を前提として、次を重視して実装する。

* Linux 上の App Service で動作可能な Node.js アプリとする
* GitHub Actions からビルド・デプロイしやすい構成にする
* 環境依存を減らし、アプリごとの差分を最小化する
* 共通パターンで管理できる実装にする
* 非 IT 職でも保守しやすいシンプルな構成にする

### 5.4 例外対応

異なるスタックが必要な場合は個別検討とする。
ただし、本標準から外れる実装は、運用・保守・デプロイの容易性を損なうため、原則として避ける。

以下は、Claude Code に与えるべき開発指示の標準である。

## 6.1 アプリ生成時の基本指示

```md
このアプリは Azure PoC プラットフォーム上で動作することを前提とする。
```

## 7. デプロイ標準フロー

### 7.1 初回セットアップ

初回セットアップはプラットフォーム管理者が担当する。
1 アプリあたりの初回セットアップは、おおむね 5～10 分を想定する。 

#### 手順

1. 開発者から GitHub リポジトリ URL を受け取る
2. Bicep テンプレート、または Azure Portal により Web App を作成する
3. Deployment Center で GitHub リポジトリと接続する
4. GitHub Actions を自動生成する
5. アプリ URL を開発者へ共有する

### 7.2 日常のデプロイ

日常の開発・反映は、開発者自身が行う。 

#### 手順

1. Claude Code でアプリを作成または修正する
2. `git push origin main` を実行する
3. GitHub Actions により自動デプロイされる
4. `https://アプリ名.azurewebsites.net` で動作確認する
5. 問題があれば `git revert` 後に再度 push し、前の安定版へ差し戻す

## 8. Azure デプロイ前提のアプリ設計ルール

この章は、Claude Code が出力するアプリを Azure App Service 向けに合わせるための設計ルールである。

## 8.1 必須要件

* Azure App Service 上で実行可能であること
* Node.js ベースでビルド・起動できること
* GitHub Actions を用いた継続デプロイに適していること
* 環境ごとの差し替えがしやすいこと
* 必要に応じて Cosmos DB / Blob Storage を接続できること
* 認証は Easy Auth を前提にできること

## 8.2 推奨要件

* 実装をシンプルに保つ
* 機能を PoC に必要な範囲へ絞る
* 開発者が `git push` 中心で運用できるようにする
* トラブル時に切り戻ししやすい構成にする
* 他アプリと同じ構成パターンを維持する

## 8.3 非推奨

* 個別最適すぎる独自構成
* 学習コストの高い複雑なフレームワーク構成
* 共有基盤での運用が難しい重い構成
* 標準外の認証方式を前提にした作り
* 例外的なランタイム前提の構成

---

## 9. 標準運用モデル

### 9.1 リソース構成

Azure 上では、次のようなリソース構成を基本とする。 

```text
Subscription
  ├ RG: poc-platform      ← App Service Plan, Cosmos DB, Blob Storage
  ├ RG: poc-project-a     ← 必要に応じてプロジェクト固有リソース
  └ RG: poc-project-n
```

### 9.2 RBAC

役割分担は次の通りとする。 

| 役割          | 権限                                         | 対象者           |
| ----------- | ------------------------------------------ | ------------- |
| プラットフォーム管理者 | App Service Plan / Cosmos DB / Storage の管理 | IT 担当 1～2 名   |
| 開発者         | 自分の Web App の閲覧・ログ確認                       | 非 IT 職 約 20 名 |

### 9.3 ライフサイクル管理

PoC アプリは増減するため、定期棚卸しを前提とする。 

1. 全 Web App に作成日タグを付与する
2. 定期的に棚卸しを実施する
3. 一定期間アクセスのないアプリは削除候補とする
4. 恒久利用価値があるものは正式環境を別途検討する

## 10. リスクと対策

本共有基盤では、以下のリスクを考慮する。 

| リスク       | 影響               | 対策                            |
| --------- | ---------------- | ----------------------------- |
| ノイジーネイバー  | 1 アプリの重い処理が他に影響  | App Insights で監視し、問題アプリを特定・制限 |
| Plan 容量不足 | アプリ増加で性能低下       | P1v3 へスケールアップ、または Plan 分割     |
| 管理者ボトルネック | Web App 作成待ち     | Bicep テンプレートで自動化              |
| コスト超過     | Cosmos DB の想定外利用 | 請求アラート設定、月次レビュー               |

### 10.1 Claude Code 実装上の注意

Claude Code は、共有基盤であることを踏まえて、次を意識して実装する。

* 不要に重い処理を避ける
* PoC に見合ったシンプルな構成にする
* 共通基盤へ負荷を集中させない
* 必要以上に DB やストレージを使わない

## 14. 結論

社内ポリシーとして **データ国内保持** と **学習利用禁止** を満たす必要がある以上、PoC 用の簡易アプリ基盤としては Azure App Service 共有構成が最も合理的である。
また、Claude Code によるアプリ生成を成功させるには、技術スタックとデプロイ前提を明確に固定し、**React + Vite / Node.js + Express / Azure App Service** を中心とした標準化を徹底することが重要である。 

本ドキュメントに従うことで、Claude Code はこのデプロイ環境に最適化されたアプリケーションを作成し、最終的に Azure の社内プラットフォーム上へデプロイ可能な形へ揃えることができる。
